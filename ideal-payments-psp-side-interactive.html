iDEAL vs Card Payments vs SEPA — Architecture Comparison

PUSH vs PULL · REDIRECT vs IFRAME · REAL-TIME vs D+5 · SCA AT BANK vs 3DS · CHARGEBACK vs IRREVOCABLE

💡 Key architectural insight: iDEAL is a push payment with a redirect. Unlike card payments where your backend calls an API and gets a synchronous result, iDEAL requires coordinating state across browser redirects, async webhooks, and potential race conditions. Design accordingly.

iDEAL

  • Push payment — customer sends funds
  • Redirect-based — customer leaves merchant site
  • SCA at issuing bank — PSD2 compliant natively
  • No chargebacks — irrevocable once authorised
  • Near-real-time settlement (SCT Inst, iDEAL 2.0)
  • Race condition: webhook vs redirect (design for it)
  • Netherlands only (expanding via iDEAL 2.0)

Card Payments

  • Pull payment — merchant pulls from card
  • Iframe / API — customer stays on merchant site
  • 3DS2 SCA — merchant-side challenge flow
  • Chargebacks possible — up to 120 days
  • Near-real-time auth, T+2 settlement
  • Synchronous auth response (simpler state machine)
  • Global acceptance

SEPA Direct Debit

  • Pull payment — creditor pulls from debtor
  • No redirect — background batch processing
  • No per-transaction SCA — mandate-based auth
  • Core: 8-week consumer refund right
  • D+1 to D+5 settlement
  • Complex: mandate lifecycle, R-transactions, batch XML
  • Pan-European SEPA zone

When to Use iDEAL

  • Customers are primarily in the Netherlands
  • One-off purchases (not recurring billing)
  • Guaranteed funds required (no chargeback risk)
  • High-trust checkout experience for NL market
  • iDEAL 2.0: recurring mandates now possible
  • Avoid for: subscriptions (use SEPA DD), international (use card)
